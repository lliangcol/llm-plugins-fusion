项目深度分析

架构结构

仓库当前是单主插件结构：根目录保存 marketplace、schema、脚本和维护文档，nova-plugin/ 保存插件 manifest、commands、skills、agents、packs、hooks 和用户文档。该结构对当前一个生产插件是清晰且低成本的，但未来新增真实插件时，目录和 release cadence 会成为主要决策点。

命令与技能映射

当前 20 个 nova-plugin/commands/*.md 与 20 个 nova-plugin/skills/nova-*/SKILL.md 保持一对一。command 是 thin slash wrapper，skill 是行为事实源，scripts/lint-frontmatter.mjs 已覆盖 frontmatter、工具权限、风险等级、共享安全策略引用和映射一致性。

验证体系

node scripts/validate-all.mjs 是总入口，覆盖 schema、registry fixture、Claude 兼容、frontmatter、agent、pack、hook config 和 docs。它在 Windows 无 Bash 时 warning skip hook shell syntax，这个设计合理，但也意味着本地验证输出需要明确区分 passed 与 skipped。

文档组织

文档按用户、插件作者、维护者和 release 角色分层：根 README 面向入口，nova-plugin/docs/ 面向插件用户和命令文档，docs/marketplace/ 面向 registry、trust 和 author workflow，docs/releases/ 面向 release hygiene。当前文档链接验证较强，但版本支持范围、旧计划标签这类“语义陈旧”还可以通过更细的规则覆盖。

发布流程

release workflow 在 tag 推送后执行 pre-release validation，并检查 tag version 必须匹配 plugin.json。手工 runbook 记录详细步骤。当前发布仍偏手工，但对单插件维护是可接受的。

插件元数据

plugin.json 保存插件自有元数据，registry.source.json 保存 marketplace 与 repository-local metadata，生成的 marketplace.json 保持 Claude-compatible，marketplace.metadata.json 和 catalog 保存 trust/risk/compatibility/review evidence。该边界正确，且 fixture 已覆盖多 entry 生成。

维护成本

主要成本来自多语言文档同步、20 组 command/skill/docs 三件套、generated outputs、Windows/Bash 差异，以及 release 手工步骤。当前脚本覆盖了结构一致性，但对“内容事实”仍依赖人工 review。

长期演进空间

短期应优先强化低风险验证补盲和事实源生成。中期可以增强 registry catalog 和 command/skill 摘要生成。长期只有在真实多插件压力出现后，才应启动 v3.0.0 多插件目录或 public portal。

方案一：验证体系补盲

推荐优先级：高复杂度：低到中风险：低

优化目标

把本轮人工发现的“当前版本支持范围”和“过期计划标签”纳入自动校验，减少发布后活跃文档事实漂移。

具体改动范围

  • 扩展 scripts/validate-docs.mjs，从 plugin.json 推导当前 minor，例如 2.1.x，校验 SECURITY.md 支持范围。
  • 为 active docs 增加 stale plan pattern，例如旧 future labels、过期 release candidate 文字、已完成版本仍作为未来计划的表述。
  • 在 docs/releases/release-hygiene.md 和 CONTRIBUTING.md 中补充该校验说明。

预期收益

直接降低发布后文档事实漂移；CI 可以在 PR 阶段拦截同类问题；维护者不需要靠全文搜索记忆。

风险与代价

正则规则过严可能误伤历史文档或 changelog。需要明确排除 archive、history 和 release notes 中的历史语境。

实施复杂度

低到中。主要是 Node 脚本规则、少量文档更新和测试样例。

回滚方式

回退 scripts/validate-docs.mjs 中新增规则和配套文档即可，不影响插件运行。

需要运行的验证命令

node scripts/validate-docs.mjs, node scripts/validate-all.mjs, git diff --check

影响

用户看到的安全支持范围、版本边界和计划状态更可信。
维护者减少人工搜索压力，但需要处理少量可能的 false positive。
CI增加 docs validation 覆盖，无新增依赖。
Release发布前能更早发现版本文字漂移，降低 patch 后补修正文档的概率。

方案二：文档事实源收敛

推荐优先级：中复杂度：中风险：中

优化目标

减少 README、overview、guides、AGENTS、CLAUDE、roadmap、release docs 中重复维护的 counts、version、date、command list 和 validation list。

具体改动范围

  • 新增一个轻量 manifest，例如 docs/project-facts.json 或复用现有 registry/source 数据生成文档片段。
  • 生成 command/skill/agent/pack count 表、质量门命令列表、当前版本/last-updated 片段。
  • 让 README 和 overview 的重复表格尽量引用生成片段或由脚本校验一致。
  • 保持 AGENTS.md 与 CLAUDE.md 的人工可读内容，但校验关键 facts 是否同步。

预期收益

降低文档同步成本；减少同一事实在多处遗漏更新；为未来多插件状态页或 catalog 扩展打基础。

风险与代价

生成文档片段可能降低文档编辑体验；如果生成边界设计过重，会把当前简洁文档流程复杂化。

实施复杂度

中。需要设计事实源、生成或校验策略，并更新维护文档。

回滚方式

删除新增生成脚本或校验规则，保留现有手工文档即可。若使用生成片段，回滚时需要恢复对应 Markdown。

需要运行的验证命令

node scripts/validate-docs.mjs, node scripts/validate-all.mjs, node scripts/generate-registry.mjs, git diff --check

影响

用户用户文档中的 counts 和版本信息更一致。
维护者长期维护成本降低，但短期需要适应生成边界。
CIdocs validation 可能增加若干一致性检查，无需新增运行时依赖。
Releaserelease 前同步步骤更可靠，但生成步骤必须写入 runbook。

方案三：registry 与 catalog 演进

推荐优先级：中复杂度：中到高风险：中

优化目标

在不启动 v3.0.0 目录迁移和不引入 frontend portal 的前提下，让 generated catalog 更像可审查的 marketplace 索引。

具体改动范围

  • 扩展 scripts/generate-registry.mjs 的 catalog 输出，加入 command count、skill count、agent count、pack count、hook presence 和 validation evidence 摘要。
  • 扩展 fixtures/registry/multi-plugin/ 覆盖新增 catalog 字段，防止多 entry 生成回归。
  • 在 docs/marketplace/catalog.md 中展示 health summary，但保持生成文件不手工编辑。
  • 必要时扩展 metadata schema，仅保留 repository-local 字段，不写入 Claude-compatible marketplace manifest。

预期收益

catalog 对插件用户和 reviewer 更有信息密度；为未来多插件需求提供真实证据，而不是提前迁移目录。

风险与代价

生成逻辑变复杂后需要更多 fixture 维护。若字段定义不稳，可能造成 catalog churn。

实施复杂度

中到高。涉及生成脚本、fixture、schema、catalog 和 docs。

回滚方式

回退生成脚本、fixture 和 schema 变更，重新运行 node scripts/generate-registry.mjs --write 恢复旧 catalog。

需要运行的验证命令

node scripts/generate-registry.mjs --write, node scripts/validate-schemas.mjs, node scripts/validate-registry-fixtures.mjs, node scripts/validate-claude-compat.mjs, node scripts/validate-docs.mjs, node scripts/validate-all.mjs

影响

用户catalog 更便于判断插件成熟度和兼容前置条件。
维护者review 信息更集中，但生成逻辑维护成本上升。
CIregistry fixture 和 schema 检查更重要，执行时间变化很小。
Releaserelease diff 会包含更丰富的 generated catalog，review 需要区分源文件与生成文件。

方案四：release 与 hook 可移植性强化

推荐优先级：中低复杂度：中风险：低到中

优化目标

降低 Windows 本地维护时 Bash 不可用造成的验证不确定性，同时让 release 前检查更清楚地区分 local skipped 与 CI passed。

具体改动范围

  • 增加 scripts/validate-environment.mjs 或在 validate-all 中输出更结构化的 prerequisites summary。
  • 为 hook 脚本增加 minimal POSIX syntax smoke fixture，CI 继续用 Bash 执行。
  • 在 release hygiene 中增加“本地 skipped 不能作为 release passed”的机器可读记录建议。
  • 评估是否为 Windows 维护者提供可选 PowerShell 辅助命令，但不替代真实 hook Bash runtime。

预期收益

维护者更容易判断本地验证可信度；release 记录不会误把 skipped hook syntax 当作 passed。

风险与代价

如果做成跨平台 hook 重写，会扩大变更面。建议仅增强环境报告和文档，不改变 hook runtime。

实施复杂度

中。脚本本身不难，关键是避免引入与 Claude Code hook 执行模型不一致的替代路径。

回滚方式

移除新增环境检查或辅助输出，保留现有 validate-all skip 行为。

需要运行的验证命令

node scripts/validate-all.mjs, node scripts/validate-hooks.mjs, bash -n nova-plugin/hooks/scripts/pre-write-check.sh, bash -n nova-plugin/hooks/scripts/post-audit-log.sh, git diff --check

影响

用户插件行为不变，但 Windows 前置条件说明更清楚。
维护者本地环境问题更早暴露，release 记录更准确。
CI继续承担真实 Bash syntax gate，可能增加环境摘要输出。
Releaserelease operator 更难误报 hook syntax passed，审计记录更可靠。

推荐结论

推荐优先执行方案一：验证体系补盲。本轮 review/fix 实际发现的问题都属于活跃文档事实漂移，而现有验证没有覆盖这些语义。把这类问题纳入 validate-docs 是最小、最直接、风险最低的改进。

方案二暂不作为首选，因为它需要先设计文档事实源边界，过早生成化可能增加维护流程复杂度。方案三暂不作为首选，因为当前 roadmap 明确 v3.0.0 仍 deferred，catalog 增强应等真实 marketplace 信息需求更明确。方案四暂不作为首选，因为当前 CI 已覆盖 Bash syntax，本地 skipped 的主要问题是记录与沟通，可以在方案一之后用较小增量处理。

建议顺序：先做方案一，再根据后续维护痛点选择方案二或方案四；方案三只在新增真实插件、catalog review 压力上升或 v3 readiness evidence 变强时启动。

风险说明

  • 方案二、方案三和方案四仍未执行，不能把这些未执行方案里的验证命令视为已通过。
  • 当前本地环境没有 Bash，本轮无法本地执行 hook shell syntax 检查；该限制已记录，CI/Linux 仍覆盖。
  • 任何修改 generated outputs 的方案都必须先改事实源，再运行 node scripts/generate-registry.mjs --write。
  • 任何新增 command、skill、agent 或 pack 的方案都必须同步 README、docs、CHANGELOG、AGENTS 和 CLAUDE 中的事实与计数。
  • 任何多插件目录、public portal 或安装路径变更都可能触发 v3.0.0 级别决策，不应在没有用户确认时执行。
