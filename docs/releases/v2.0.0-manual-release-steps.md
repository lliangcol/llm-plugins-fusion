## Do Not Start A Future Manual Release If

- The working tree contains unreviewed changes outside the intended release
  docs or release metadata.
- `.codex/` or other local runtime artifacts appear in `git status`.
- `node scripts/validate-all.mjs` fails.
- `node scripts/validate-all.mjs` reports skipped hook syntax checks and Bash is
  expected to be available for the release operator.
- The target GitHub repository is unclear.
- A `v2.0.0` tag already exists locally or remotely and has not been verified as
  the intended release tag.

## 1. Confirm Repository Target

Confirm the remote points to the repository that should own the public release:

```powershell
git remote -v
git branch --show-current
```

If the remote is not the intended public repository, stop and correct the remote
or perform the release from the correct checkout. Do not push the release tag to
an uncertain repository.

Also confirm the public repository slug used by `README.md`, plugin metadata,
and the Claude Code marketplace install command is the same repository that will
receive the tag. If these disagree, resolve the repository target before
publishing.

## 2. Final Local Validation

Use Node.js 20 or newer. On Windows, ensure Git Bash, WSL Bash, or another Bash
runtime is available on `PATH` before running the full validation suite.

PowerShell example when Git Bash is installed but not already on `PATH`:

```powershell
$env:PATH = "C:\Path\To\Git\bin;$env:PATH"
```

Run the final checks:

```powershell
node --version
node scripts/generate-registry.mjs --write
node scripts/validate-all.mjs
git diff --check
git status --short
```

Expected results:

- `validate-all` ends with `Summary: failed=0 skipped=0`.
- `generate-registry` does not leave unexpected generated-file diffs.
- `git diff --check` prints no whitespace errors.
- `git status --short` contains only the release changes that will be committed.

## 3. Review The Release Diff

Inspect the complete diff before committing:

```powershell
git diff
```

For this release, expected tracked release-prep files include:

- `CHANGELOG.md`
- `ROADMAP.md`
- `README.md`
- `docs/releases/vnext-release-decision.md`
- `docs/releases/v2.0.0-manual-release-steps.md`

The exact file list may differ if another maintainer has already committed some
of these changes. Do not include `.codex/`, temporary logs, editor files, or
machine-local artifacts.

## 4. Commit Release Preparation

Stage only the intended release files:

```powershell
git add CHANGELOG.md ROADMAP.md README.md docs/releases/vnext-release-decision.md docs/releases/v2.0.0-manual-release-steps.md
git status --short
```

If the staged set is correct, commit:

```powershell
git commit -m "docs: document v2.0.0 release validation"
```

Run the release validation one more time after the commit:

```powershell
node scripts/validate-all.mjs
```

Expected result:

```text
Summary: failed=0 skipped=0
```

## 5. Land On Main

If working directly on `main`, push the commit:

```powershell
git push origin main
```

If working on a feature branch, push the branch and merge it through the normal
review path:

```powershell
git push origin HEAD
```

After the merge, update local `main`:

```powershell
git fetch origin
git switch main
git pull --ff-only origin main
```

Do not create the release tag from an unmerged feature branch unless the
maintainer intentionally wants that branch commit to become the release commit.

## 6. Verify Version And Existing Tags

Confirm the checked-out commit contains the intended version:

```powershell
node -e "process.stdout.write(require('./nova-plugin/.claude-plugin/plugin.json').version)"
```

Expected output:

```text
2.0.0
```

Check local and remote tags:

```powershell
git fetch --tags origin
git tag --list v2.0.0
git ls-remote --tags origin v2.0.0
```

If either command shows an existing `v2.0.0` tag, stop and inspect it before
continuing:

```powershell
git show --stat v2.0.0
```

Do not overwrite a public release tag unless the maintainer explicitly decides
to replace a bad release.

## 7. Create And Push The Release Tag

Create an annotated tag from the validated `main` commit:

```powershell
git tag -a v2.0.0 -m "nova-plugin v2.0.0"
```

Confirm the tag points at the intended commit:

```powershell
git show --stat v2.0.0
```

Push the tag:

```powershell
git push origin v2.0.0
```

This tag push triggers `.github/workflows/release.yml`.

## 8. Monitor GitHub Release Workflow

In GitHub Actions, open the `Release` workflow run triggered by tag `v2.0.0`.
The workflow must pass both jobs:

- `Pre-release Validation`
- `Create GitHub Release`

If GitHub CLI is available, use:

```powershell
gh run list --workflow Release --limit 5
gh run watch <run-id>
gh release view v2.0.0
```

Expected release name:

```text
nova-plugin v2.0.0
```

The release body should be generated from `CHANGELOG.md` section `2.0.0`.

## 9. Post-release Smoke Checks

After the release workflow passes, verify the published state:

```powershell
git ls-remote --tags origin v2.0.0
gh release view v2.0.0
```

In Claude Code, confirm marketplace install still works from the public
repository:

```bash
/plugin marketplace add lliangcol/llm-plugins-fusion
/plugin install nova-plugin@llm-plugins-fusion
```

Then confirm the plugin exposes the expected user-facing surface:

- 20 commands are available.
- `/review`, `/review-lite`, `/review-only`, and `/review-strict` remain
  available.
- The active agent documentation describes the 6-core model.
- The release notes clearly state that active-agent changes are breaking.

## 10. Failure Handling

If validation fails before the tag is pushed:

1. Do not create the tag.
2. Fix the issue on the release branch.
3. Re-run `node scripts/validate-all.mjs`.
4. Commit the fix.
5. Restart from the tag verification step.

If the tag was created locally but not pushed:

```powershell
git tag -d v2.0.0
```

If the tag was pushed but the release workflow failed:

1. Inspect the failed workflow log.
2. If the failure is caused by repository content, commit a fix to `main`.
3. Decide whether to delete and recreate the tag or publish a follow-up patch
   release. For a public release tag, prefer a patch release once users may have
   seen the tag.

Only delete a pushed tag when the maintainer accepts that the original tag must
not remain public:

```powershell
git push origin :refs/tags/v2.0.0
git tag -d v2.0.0
```

Then recreate the tag from the corrected commit:

```powershell
git tag -a v2.0.0 -m "nova-plugin v2.0.0"
git push origin v2.0.0
```
